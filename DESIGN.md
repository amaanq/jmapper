# Design notes — non-obvious choices & divergences from jmap-perl

## Problem shape

jmap-perl solves: translate IMAP (a stateful, UID-based, per-folder protocol)
into JMAP (a stateless, state-string-versioned, object-graph protocol). Its
canonical move is a **two-layer cache** (SQLite there, PostgreSQL here):

- **IMAP cache** (`ifolders`, `imessages`, `iserver`) — raw per-folder state.
- **JMAP projection** (`jmailboxes`, `jmessages`, `jmessagemap`, `jthreads`,
  `jrawmessage`) — denormalized, client-facing.

State is tracked by per-row `jmodseq` counters bumped on every write, so
`Email/changes(sinceState=N)` compiles to `WHERE jmodseq > N`.

jmap-rust keeps the two-layer shape but flattens a couple of details (see
below). Critically: IMAP is the source of truth; the cache is reconciled from
it, never the other way around.

### 0. PostgreSQL + cornucopia, not SQLite + sqlx

The original port used per-daemon SQLite. Storage moved to PostgreSQL: the
deployment target already runs one postgres for everything, and the typed
query layer is now generated by cornucopia from `queries/*.sql`, validated
at generation time against a live database initialized from `schema.sql` and
the checked-in `crates/jmapper-codegen` crate. Genuinely dynamic SQL — the
Email/query filter compiler, sort-variant queries — still lives in Rust and
runs through raw tokio-postgres with `$N` params; a router test executes
the full filter/sort vocabulary against real postgres so dialect drift
fails loudly. Timestamps stay BIGINT unix seconds end to end: chrono
converts at the edges and the sync loop compares raw IMAP INTERNALDATE
seconds.

## Choices worth calling out

### 1. msgid is `sha1(normalized Message-ID)`, not X-GM-MSGID

**Divergence from jmap-perl**, which uses Gmail's `X-GM-MSGID` extension to
get a stable id across labels.

Rationale: using `sha1(RFC 5322 Message-ID)` works identically for Gmail and
generic IMAP. The same message-id across folders collapses to the same msgid
naturally, and many-to-many membership ends up in `message_mailboxes` without
needing X-GM-LABELS. Threading uses `X-GM-THRID` when the server advertises
`X-GM-EXT-1` (async-imap parses the attribute but exposes no accessor, so
`fetch_gmail_thrids` reads the untagged FETCH replies off the public
unsolicited-responses channel) and RFC 5322 References/subject heuristics
otherwise; both record into the same `thread_index` so mixed sources
converge.

### 2. Rustls + ring, never OpenSSL

**Divergence**: jmap-perl uses `Net::SSL` (OpenSSL).

Rationale: NixOS and many minimal container images ship without system
OpenSSL, and cross-compiling `-sys` crates is painful. We route all TLS
through `tokio-rustls` with the `ring` backend. `reqwest` (used by OAuth and
DAV) is pinned to `rustls-tls` for the same reason. The whole release binary
is openssl-free.

### 3. IDLE only, no polling fallback

**Divergence**: jmap-perl's `Email/changes` implicitly polls IMAP with
`STATUS` on every JMAP request.

Rationale: in 2026 every IMAP server worth targeting supports IDLE. Polling
is wasted bandwidth and latency; IDLE gives near-real-time updates. We probe
`CAPABILITY` at startup and refuse to start an account whose server lacks
IDLE, which surfaces a broken config loudly instead of silently degrading.

### 4. On-demand body fetch is mpsc-routed, not connection-per-request

**Divergence**: jmap-perl's `fill_messages` opens / reuses a separate blocking
connection each call.

Rationale: we own exactly one IMAP connection per account, kept in IDLE. When
the HTTP layer needs a body for `Email/get bodyValues`, it posts a
`FetchBody { msgid, respond: oneshot }` into the account task's mpsc; the
account task drops its IDLE `StopSource`, lets the wait-future resolve,
exits IDLE cleanly, performs `UID FETCH (BODY.PEEK[])`, writes to
`raw_messages`, responds, and re-enters IDLE. One connection, no pooling,
no duplicate state.

Tradeoff: concurrent body fetches are serialized per account. For a single
human driving Bulwark that is fine; if it matters later, a dedicated fetch
connection is the obvious upgrade.

### 5. Filter compilation pushes into SQL, not in-process Perl

**Divergence**: jmap-perl evaluates `Email/query` filters in Perl after
loading all candidate rows.

Rationale: the database handles `LIKE` + `received_at > $n` + indexed
`account_id` joins in microseconds for the ~20k-envelope corpora we target.
We build a small `Vec<SqlParam>` + `WHERE` string from the typed
`EmailFilter` struct and let PostgreSQL do the work. AND/OR/NOT operator
trees compile recursively into parenthesized SQL; `inMailbox` uses an
EXISTS subquery (not a JOIN) so the predicate stays a self-contained
boolean under OR/NOT.

### 6. Session state hashes the authenticated account's visible configuration

**Divergence**: jmap-perl derives session state from the server's global
modseq counter.

Rationale: the session state changes whenever the authenticated account's
Session resource could change: identity metadata, URLs, limits, or advertised
capabilities. Unrelated accounts stay out of the hash. Per-type state
(`Email/changes`'s `sinceState`) is separate and tracks data modseqs.

### 7. Bearer tokens are hashed in memory

**Divergence**: jmap-perl does HTTP basic auth with plaintext compare.

Rationale: we SHA-256-hash the configured token at startup and keep a
constant-time compare in the middleware. The plaintext never lingers in the
process image beyond startup.

### 8. `[Gmail]/Starred` and `[Gmail]/Important` are skipped

**Same as jmap-perl**. Those folders are just redundant views of `All Mail`
with an extra flag; syncing them triples our envelope cost and confuses the
mailbox list with synthetic "Starred" mailboxes.

### 9. UIDVALIDITY change = folder rebuild, not fatal

**Divergence**: jmap-perl calls `die` when `UIDVALIDITY` changes on a folder.

Rationale: `UIDVALIDITY` rotations happen when the server rebuilds a folder
or restores from backup. Killing the process is hostile; the correct
response is to drop the folder's rows and do an initial sync on it again.
Our `delta_sync` returns `UidValidityChanged` and the caller (sync loop) will
see it next tick and retry.

### 10. Live state is ArcSwap, not RwLock

**Choice**: `AppState` wraps its mutable interior (accounts plus IMAP and DAV
task handles/capability availability) in `Arc<ArcSwap<AppStateLive>>`. HTTP
handlers call `state.live()` to get a snapshot `Arc<AppStateLive>` for the
duration of their request.

Rationale: reads are the hot path, writes are exactly one place (SIGHUP
reload). An `RwLock` would serialize readers against a once-per-week writer;
ArcSwap is lock-free on the read path and swaps atomically. The snapshot the
handler holds remains valid even if a reload fires mid-request.

### 11. Config hot reload on SIGHUP (accounts only)

**Choice**: SIGHUP re-reads the TOML, hashes each account's credentialed
fields into a fingerprint, and diffs against the running task map:
removed/changed accounts get a `Shutdown` message + join wait; new and
changed accounts get a freshly spawned task. The HTTP-visible state
(`accounts`, `account_senders`) is atomically swapped.

Rationale: single-user ops wants to add a Gmail account or rotate a bearer
token without downtime. A full process restart would drop in-flight
`Email/get` fetches and IDLE state. Limiting the reload scope to the
`[[accounts]]` table keeps the reconciliation logic honest — touching
`bind` / CORS / session URL requires a real restart because rebuilding the
axum router on the fly is fiddly and the blast radius rarely justifies it.

### 12. Mailbox rename keeps the JMAP id via a folder link column

Mailbox ids are derived from `sha1(account, imap_name)` at discovery. A
RENAME through `Mailbox/set` would therefore mint a new id and orphan the
old row — breaking the JMAP contract that update preserves object identity.
The fresh schema stores `folders.mailbox_id` as the authoritative link,
written on first ingest and preserved across renames. Child folders (renamed
implicitly by the server per RFC 3501 §6.3.5) get their cached paths rewritten
in the same pass so their ids survive too. This project has no migration
history: startup initializes a new database directly from `schema.sql`.

### 13. SMTP delegates to mail-send

`EmailSubmission/set` needs XOAUTH2 with the same access token the IMAP
session refreshes and rustls-only TLS. A thin adapter configures mail-send,
validates envelope addresses, and preserves the smarthost's post-DATA reply.
The transaction runs inside the account task (serialized with IDLE).
Immediate submissions therefore succeed only after the smarthost accepts the
message; future `sendAt` submissions use the bounded in-process queue below.

### 14. Email/set create composes narrow MIME on purpose

mail-builder handles one text part, one HTML part (multipart/alternative) and
uploaded-blob attachments (multipart/mixed). An explicit
`bodyStructure` is rejected rather than approximated — silently flattening
a client-described nesting would corrupt the message. Composed bytes are
staged through the same `uploaded_blobs` row an upload would create and ride
the existing `Email/import` APPEND path, so create needed no new IMAP code.

### 15. Delayed send stages bytes, schedules in-process

A future `sendAt` stores the submission as `pending` with its own copy of
the raw message — the standard client flow destroys the draft via
`onSuccessDestroyEmail` immediately, which would CASCADE the `raw_messages`
row the send needs. A 15 s poll loop claims due rows with a compare-and-set
to `'sending'` (a concurrent cancel loses cleanly at the WHERE clause),
relays via the account task, and retries up to 5 ticks before marking the
row `canceled` — the spec's undoStatus has no "failed" value and `canceled`
at least tells the client nothing went out.

## Things NOT done (yet)

- **`header` filter on Email/query.** Arbitrary headers are neither cached
  nor indexed; matching raw bytes would false-positive on body content.
- **`body` filter completeness.** Searches lazily-cached bodies + previews;
  a full-account body mirror would defeat the lazy body cache.
- **DSN parsing / MDN (RFC 9007).** `deliveryStatus` reports the
  smarthost's accept reply (`delivered: "queued"`); anything beyond that
  needs the bounce/receipt feedback loop of a real MTA.
- **Expanded calendar query results and scheduling.** Date filters expand
  recurrence data through calcard, but `expandRecurrences` does not mint
  synthetic occurrence ids. iTIP/iMIP, managed attachments, and
  `utcStart`/`utcEnd` writes are not implemented.

## Workspace structure rationale

Five crates isolate concerns:

- `jmap-protocol` has no I/O and no async; pure serde. Keeps its tests fast
  (6 ms) and lets us swap transports / storage without touching types.
- `imap-sync` owns IMAP, OAuth, and the PostgreSQL cache. Embeddable in
  tests that don't need a network; `testkit` mints a throwaway database per
  test with the real schema.
- `dav-sync` owns the CalDAV/CardDAV HTTP and XML layers, iCalendar/vCard
  conversion, atomic PostgreSQL cache updates, and per-account DAV service.
- `jmap-server` is thin: axum router + method dispatch. Delegates queries to
  `imap-sync::db` so it never builds raw SQL itself (filter compilation
  excepted — the filter→SQL mapping is tightly bound to the message row layout
  and doesn't belong in the generic DB layer).
- `jmapper` is the only binary. Owns config, process lifecycle, and the
  OAuth bootstrap subcommand.

## Testing strategy

- **`jmap-protocol`** — serde round-trip tests for each envelope type and for
  the filter AST's `untagged` discrimination.
- **`imap-sync::db`** — typed statements are validated by cornucopia
  against the live schema at generation time; behavioral coverage lives in
  the router integration suite against per-test postgres databases.
- **`imap-sync::sync`** — pure helpers (mailbox id stability, Message-ID
  normalization, date parsing). IMAP integration tests against real accounts
  are left as a manual smoke test.
- **`imap-sync::imap`** — role derivation (inbox / sent / trash mapping).
- **`dav-sync`** — conversion unit tests plus mock-server integration tests
  for discovery, incremental/fallback sync, ETag conflicts, and writes.
- **End-to-end** — the router integration suite exercises auth, account
  isolation, method dispatch, object limits, MIME/blob behavior, mail writes,
  delayed submission, DAV-backed methods, and protocol error shapes against
  per-test PostgreSQL databases. A live IMAP/DAV provider remains a manual
  deployment smoke test.
